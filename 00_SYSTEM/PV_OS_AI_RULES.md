## 一、AI 平台角色分工

PV_OS 项目中可使用四类 AI 平台，各司其职，不可越界混用：

### 1.1 ChatGPT Plus（Web 端）

**定位：** 策略大脑 · 内容创意中心

| 维度 | 说明 |
|------|------|
| **适用场景** | 市场策略规划、Prompt 工程、爆款内容脚本创作、竞品策略分析、商业方案策划 |
| **核心优势** | 联网搜索、Canvas 长文协作、网页阅读、跨会话记忆 |
| **不可用于** | 本地文件批量管理、Shell 执行、自动化流水线、代码部署 |
| **典型指令** | "帮我分析这 20 条光伏爆款视频的共性结构"、"生成 5 条针对别墅业主的光伏科普脚本" |

### 1.2 Codex Desktop（桌面端）

**定位：** 项目总管 · 文件操作系统

| 维度 | 说明 |
|------|------|
| **适用场景** | 本地文件管理、代码编辑、目录结构调整、文档批量维护、Git 操作 |
| **核心优势** | 本地文件系统直连、MCP 工具扩展、跨文件重构、可视化操作 |
| **不可用于** | 需要联网实时搜索的场景、外部平台数据采集、Web 浏览 |
| **典型指令** | "把 02_DATA/raw/ 下的 JSON 统一迁移到按平台分目录"、"更新所有 PV_OS 目录地图" |

### 1.3 Codex CLI（终端）

**定位：** 自动化管道 · 批量执行器

| 维度 | 说明 |
|------|------|
| **适用场景** | Shell 脚本执行、数据清洗流水线、定时任务编排、系统环境配置 |
| **核心优势** | 直接操作终端、管道命令、与系统工具深度集成 |
| **不可用于** | 需要 GUI 的操作、需要联网浏览的场景、纯策略讨论 |
| **典型指令** | "批量清洗 02_DATA/raw/douyin/ 下所有 CSV"、"部署采集脚本到 crontab" |

### 1.4 DeepSeek（国产模型）

**定位：** 中文内容引擎 · 成本优化器

| 维度 | 说明 |
|------|------|
| **适用场景** | 批量中文内容生成、中文评论情感分析、成本敏感的大规模文本处理 |
| **核心优势** | 中文语义理解深入、API 成本低、适合高频批量调用 |
| **不可用于** | 需要联网搜索的场景、多模态识别、复杂代码生成 |
| **典型指令** | "对 2000 条评论区内容做潜在客户意图分类"、"批量生成 100 条光伏科普推文标题" |

### 1.5 协作原则

```
ChatGPT Plus  →  定方向、出策略、写爆款
Codex Desktop →  管文件、改代码、调结构
Codex CLI     →  跑脚本、洗数据、做调度
DeepSeek      →  产中文、做分类、降成本
```

- **不要用错平台**：不让 ChatGPT 替代 Codex CLI 跑 Shell，不让 Codex CLI 替代 ChatGPT 做策略
- **任务分级执行**：策略层（ChatGPT）→ 执行层（Codex Desktop / CLI）→ 批处理层（DeepSeek）
- **单一平台不适配时**：拆解任务到适配的平台分别执行，不做勉强适配

## 二、AI 执行流程

### 2.1 通用执行流程

```
┌──────────────────────────────────────────────────────┐
│ Step 0：进入项目                                      │
│   ├─ 读取 00_SYSTEM/PV_OS_DIRECTORY_MAP.md           │
│   ├─ 读取 00_SYSTEM/PV_OS_AI_RULES.md（本文件）       │
│   └─ 读取 00_SYSTEM/PV_OS_PROJECT_STATUS.md          │
├──────────────────────────────────────────────────────┤
│ Step 1：理解任务                                      │
│   ├─ 识别任务属于哪条核心业务线（见目录地图第七章）    │
│   └─ 定位到对应主目录                                  │
├──────────────────────────────────────────────────────┤
│ Step 2：执行操作                                      │
│   ├─ 仅操作任务相关目录，不跨域                       │
│   ├─ 新增文件前通过目录地图第六章决策树判断位置        │
│   └─ 修改现有文件前先读取理解全文                     │
├──────────────────────────────────────────────────────┤
│ Step 3：记录变更                                      │
│   ├─ 重大修改更新 PV_OS_PROJECT_STATUS.md             │
│   └─ 重要决策记录到对应目录的变更日志                 │
└──────────────────────────────────────────────────────┘
```

### 2.2 特殊场景流程

**数据采集流程：**
```
确定采集目标 → 检查 09_AI_OPERATION/platforms/ 配置 
→ 执行采集脚本 → 原始数据 → 02_DATA/raw/
→ 清洗处理 → 02_DATA/processed/
```

**客户识别流程：**
```
评论数据输入 → 03_AI_AGENT/agents/comment_analyzer 
→ 潜在客户识别 → 客户评分 → 05_CUSTOMER_CRM/leads/
```

**内容生产流程：**
```
爆款拆解（09_AI_OPERATION/）→ 脚本生成（04_CONTENT/scripts/）
→ 素材匹配（04_CONTENT/materials/）→ 成品输出（04_CONTENT/outputs/）
```

### 2.3 执行纪律

- **操作前先读目录地图**：不确定文件该放哪时，查阅 `PV_OS_DIRECTORY_MAP.md` 第六章决策树
- **最小修改原则**：每次操作仅修改与任务直接相关的文件，不顺便 "优化" 无关文件
- **先确认再执行**：对存量文件做结构性修改（重命名、移动、删除）前，先确认影响范围
- **不假设不存在的内容**：如需引用其他文件，先确认其存在

## 三、文件管理规则

### 3.1 总体原则

所有文件操作必须遵循 `PV_OS_DIRECTORY_MAP.md` 的定义，不得自创目录或混放文件。

### 3.2 命名规范

| 文件类型 | 命名规范 | 示例 |
|----------|---------|------|
| 系统文档 | `PV_OS_` 前缀，全大写蛇形 | `PV_OS_DIRECTORY_MAP.md` |
| 数据文件 | `平台_类型_日期.格式` | `douyin_comments_20260711.csv` |
| Agent 定义 | 小写蛇形 | `comment_analyzer.yml` |
| 内容脚本 | `场景_主题_版本.md` | `villa_solar_benefits_v1.md` |
| 配置文件 | 语义化命名 + `.config` 后缀 | `douyin_api.config.yml` |
| 日志文件 | `模块_日期.log` | `collector_20260711.log` |

### 3.3 文件生命周期

```
创建 → 放置到正确目录 → 在对应的索引/README 中注册（如需要）
     → 使用 → 修改（记录变更）→ 废弃 → 移至 99_BACKUP/deprecated/
```

### 3.4 禁止事项

- ❌ 在根目录创建非目录文件（所有文件必须归属到具体一级目录）
- ❌ 在数据目录（02_DATA）中混放代码或配置
- ❌ 在系统目录（08_SYSTEM）中混放业务数据
- ❌ 使用中文文件名（除内容脚本的标题性文件外）
- ❌ 重复创建相同用途的文件

## 四、业务目标约束

### 4.1 行业边界

PV_OS **仅聚焦光伏（太阳能）行业**，所有 AI 行为必须在以下范围内：

- ✅ 家庭光伏系统（并网/离网/储能）
- ✅ 小商业光伏解决方案
- ✅ 光伏内容营销与客户开发
- ❌ 风电、水电、其他新能源（除非竞品对比分析需要）
- ❌ 与光伏无关的通用营销

### 4.2 客户价值导向

所有 AI 产出必须服务于以下客户价值点：

| 价值点 | 说明 |
|--------|------|
| **省钱** | 电费节省计算、投资回报分析 |
| **环保** | 碳减排数据、绿色生活理念 |
| **品质** | 用电稳定性、储能备电安全感 |
| **面子** | 别墅光伏的科技感与品味标签 |
| **收益** | 余电上网收益、政策补贴 |

### 4.3 数量化思维

- AI 输出涉及收益、成本、省电时，必须附带**可量化数据**
- 不确定的数据标注 "预估" 或 "参考值"
- 引用政策补贴时注明时效性

## 五、客户开发方向约束

### 5.1 三条核心客群线

| 序号 | 客群 | 典型场景 | 关键词 |
|------|------|---------|--------|
| 1 | **城市家庭** | 普通住宅光伏安装 | 省钱、回本、一站式 |
| 2 | **高端住宅** | 别墅、叠拼、花园洋房、露台、阳光房 | 品质、美观、定制化、储能 |
| 3 | **小商业** | 民宿、酒店、棋牌室、茶楼、美容院 | 省电费、提升档次、吸引客户 |

### 5.2 客群优先策略

- **内容优先级**：高端住宅 > 小商业 > 城市家庭（按客单价排序）
- **采集优先级**：三线并行，根据数据量动态调整
- **话术差异化**：不同客群使用不同的利益点话术（家庭讲省钱，别墅讲品质，商业讲增收）

### 5.3 禁止开发方向

- ❌ 大型工商业光伏（非当前阶段目标）
- ❌ 光伏组件生产/供应链（非运营核心）
- ❌ 地面电站/集中式光伏（业务不匹配）

## 六、数据采集原则

### 6.1 合规底线

- **仅采集公开数据**：不绕过登录墙、不破解接口、不爬取私密内容
- **遵守平台规则**：各平台的 robots.txt 和用户协议优先
- **个人信息脱敏**：采集的评论用户信息做脱敏处理后再存储

### 6.2 采集质量标准

| 维度 | 要求 |
|------|------|
| **完整性** | 每条数据包含时间戳、来源平台、原始链接 |
| **结构化** | 原始 → 02_DATA/raw/，清洗后 → 02_DATA/processed/，保留处理链路 |
| **可追溯** | 每次采集记录采集时间、采集参数、数据量 |
| **不重复** | 采集前检查去重，避免存量数据重复入库 |

### 6.3 平台覆盖

| 平台 | 采集内容 | 优先级 |
|------|---------|--------|
| 抖音 | 视频、评论区、账号信息 | 🔴 最高 |
| 小红书 | 笔记、评论区、账号信息 | 🔴 最高 |
| 快手 | 视频、评论区 | 🟡 中 |
| 视频号 | 视频、互动数据 | 🟡 中 |

## 七、竞品账号分析原则

### 7.1 分析维度

对每个竞品账号，必须覆盖以下分析维度：

1. **账号基本面**：粉丝量、作品量、互动率、更新频率
2. **内容策略**：选题方向、内容形式、人设定位
3. **爆款规律**：高互动内容的共性特征（时长、话题、形式）
4. **客户转化**：评论区是否有明确购买意向、如何引导私域
5. **可复制性**：哪些策略 PV_OS 可直接借鉴

### 7.2 分析纪律

- 竞品分析报告存放在 `09_AI_OPERATION/competitors/`
- 每次分析标注日期，定期更新（月度刷新）
- 不照搬竞品内容，做策略提炼而非像素级模仿

## 八、内容生产原则

### 8.1 内容策略金字塔

```
          ┌─────────┐
          │  品牌层  │  ← 建立 PV_OS 行业权威
          ├─────────┤
          │  信任层  │  ← 真实案例、客户评价、施工实拍
          ├─────────┤
          │  种草层  │  ← 省钱计算、政策解读、科普内容
          ├─────────┤
          │  引流层  │  ← 热点话题、反差内容、互动话题
          └─────────┘
```

### 8.2 内容生产规范

| 规范项 | 要求 |
|--------|------|
| **真实性** | 案例数据可验证，不编造客户故事 |
| **专业性** | 涉及光伏技术参数时必须准确，不确定的标注 "参考值" |
| **平台适配** | 抖音重前 3 秒钩子，小红书重封面+清单体，快手重真实感，视频号重社交传播 |
| **AI 标识** | AI 生成内容在内部标注引擎和版本，发布前人工审核 |

### 8.3 爆款拆解规则

- 每篇拆解报告存放于 `04_CONTENT/viral_analysis/`
- 必须拆解出：**钩子公式、内容结构、情绪曲线、可复用模板**
- 拆解结果反哺到 `03_AI_AGENT/prompts/` 的内容生成模板

## 九、AI 执行偏差纠正机制

### 9.1 偏差类型与纠正

| 偏差类型 | 表现 | 纠正动作 |
|----------|------|---------|
| **目录错位** | 文件放到错误目录 | 立即移动到正确位置，更新相关索引 |
| **范围蔓延** | 擅自修改无关文件 | 回滚无关修改，重新限定任务范围 |
| **格式违规** | 文件命名不符合规范 | 重命名，输出格式提醒 |
| **数据污染** | 将未清洗数据放入 processed/ | 移回 raw/，已污染的标记待重新清洗 |
| **内容虚构** | 编造案例数据或客户信息 | 删除虚构内容，标注为"待验证数据" |
| **竞品照搬** | 直接复制竞品内容 | 删除，改为策略提炼格式 |

### 9.2 自检机制

每次 AI 完成操作后，静默执行以下自检：

1. ✏️ 新建/修改的文件是否在正确的目录？
2. ✏️ 文件名是否符合规范？
3. ✏️ 是否误操作了任务范围外的文件？
4. ✏️ 生成的内容是否包含虚构信息？

### 9.3 纠正流程

```
检测到偏差 → 立即停止当前操作
          → 分析偏差原因（规则不清 / 错误判断 / 疏忽）
          → 执行纠正动作（移动/重命名/删除/回滚）
          → 记录到 00_SYSTEM/PV_OS_PROJECT_STATUS.md
          → 如为规则不清导致，补充本文件相关条款
```

## 十、变更记录规则

### 10.1 必须记录的事项

以下情况必须在 `PV_OS_PROJECT_STATUS.md` 中记录：

- 一级目录的新增、删除、重命名
- 核心文件的创建或重大修改（超过 50% 内容变更）
- 数据采集管道或配置的重大变更
- Agent 角色定义的新增或废弃
- AI 协作规则的版本升级

### 10.2 记录格式

```markdown
| 日期 | 变更类型 | 变更内容 | 影响范围 | 操作者 |
|------|---------|---------|---------|--------|
| 2026-07-11 | 新增 | 创建 PV_OS_DIRECTORY_MAP.md V1.0 | 全局 | Codex |
```

### 10.3 版本号规则

- 所有 `00_SYSTEM/` 下的规则文件使用语义化版本号
- **主版本号**：架构性变更（如目录结构重大调整）
- **次版本号**：功能新增或规则扩展
- **修订号**：文字修正、格式调整
