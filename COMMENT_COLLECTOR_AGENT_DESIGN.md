## 一、Agent 定位

### 1.1 PV_OS 完整链路中的位置

```
视频平台（抖音 / 小红书 / B站 / 快手 / 视频号）
    │
    ▼
┌─────────────────────────────────────────────────┐
│  comment_collector_agent（本文档定义）              │  ← P0，当前缺失
│  职责：连接平台 → 发现账号 → 发现视频 → 采集评论  │
│  输出：02_DATA/raw/（comment_schema 标准格式）     │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  数据清洗（08_SYSTEM/scripts/data_cleaner.py）     │  ← 当前空脚本
│  职责：去重、去噪、格式校验                          │
│  输出：02_DATA/04_COMMENT_DATABASE/               │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  comment_intent_model（PV_OS_COMMENT_INTENT_MODEL.md）│
│  职责：语义理解 → 意图分类 → 客户识别                │
└─────────────────────────────────────────────────┘
    │
    ▼
comment_analyzer（03_AI_AGENT/agents/comment_analyzer/）
    │
    ▼
lead_scoring_agent（03_AI_AGENT/agents/lead_scoring_agent/）
    │
    ▼
CRM（05_CUSTOMER_CRM/）
```

### 1.2 定位说明

`comment_collector_agent` 是 PV_OS 链路的最前端，唯一的数据入口。没有这个 Agent，下游所有模块（意图模型、comment_analyzer、lead_scoring_agent、CRM）都无法运转。

| 属性 | 说明 |
|------|------|
| **角色** | 数据生产者 |
| **上游** | 视频平台（外部） |
| **直接下游** | 数据清洗 → comment_intent_model |
| **核心原则** | 采集原始评论，不做价值判断——把"看懂"交给下游 AI |

## 二、平台采集策略

### 2.1 五平台价值判断

以下判断基于各平台公开特征与 PV_OS 城市小区客户定位推导，具体数据需待采集验证。

#### 抖音

| 维度 | 评级 | 判断 |
|------|:--:|------|
| 城市用户比例 | ⭐⭐⭐⭐⭐ | 全人群覆盖，川渝黔城市家庭用户基数最大 |
| 光伏内容密度 | ⭐⭐⭐⭐⭐ | 大量本地安装商运营，安装实拍类视频集中 |
| 真实客户评论概率 | ⭐⭐⭐⭐ | 评论直接，"多少钱""怎么装""联系我"高频 |
| 商业意向强度 | ⭐⭐⭐⭐⭐ | 看到本地案例后"我家能不能装"需求最直接 |
| 数据获取难度 | ⭐⭐⭐ | 有 API 但评论数据需爬虫/第三方工具 |

**采集优先级：P0（最高）**

> 引用：`COMMENT_ANALYZER_RULE.md` §一 评论来源定义。

#### 小红书

| 维度 | 评级 | 判断 |
|------|:--:|------|
| 城市用户比例 | ⭐⭐⭐⭐⭐ | 一线/新一线城市女性为主，消费力强 |
| 光伏内容密度 | ⭐⭐⭐ | 光伏内容以阳光房改造/家庭能源分享为主，非密集 |
| 真实客户评论概率 | ⭐⭐⭐⭐ | "求推荐""多少钱""哪里做的"自然且高频 |
| 商业意向强度 | ⭐⭐⭐⭐ | 习惯主动询价/要推荐，商业意图自然 |
| 数据获取难度 | ⭐⭐ | 反爬严格，技术门槛高 |

**采集优先级：P1（高）**

#### B站

| 维度 | 评级 | 判断 |
|------|:--:|------|
| 城市用户比例 | ⭐⭐⭐⭐ | 一二线城市年轻用户为主 |
| 光伏内容密度 | ⭐⭐ | 以科普/评测为主，安装类内容少 |
| 真实客户评论概率 | ⭐⭐⭐ | 评论以讨论技术参数为主 |
| 商业意向强度 | ⭐⭐ | "了解技术"和"比较方案"为主，购买意图密度较低 |
| 数据获取难度 | ⭐⭐⭐⭐ | API 相对开放 |

**采集优先级：P1（中高，长期培育价值）**

#### 快手

| 维度 | 评级 | 判断 |
|------|:--:|------|
| 城市用户比例 | ⭐⭐ | 下沉市场为主，城市用户占比低于抖音/小红书 |
| 光伏内容密度 | ⭐⭐⭐ | 农村自建房光伏为主，城市小区客户密度较低 |
| 真实客户评论概率 | ⭐⭐⭐ | 评论直接，但城市相关度低 |
| 商业意向强度 | ⭐⭐⭐ | "多少钱""怎么装"类常见 |
| 数据获取难度 | ⭐⭐⭐ | 与抖音同体系，技术路径相似 |

**采集优先级：P2（低，仅采集城市+经营场景）**

> 引用：`CUSTOMER_SCORE_MODEL.md` §8.3（农村经营场景按商业客户处理）。

#### 视频号

| 维度 | 评级 | 判断 |
|------|:--:|------|
| 城市用户比例 | ⭐⭐⭐ | 微信生态，全年龄段 |
| 光伏内容密度 | ⭐⭐ | 公众号关联视频为主，评论生态弱 |
| 真实客户评论概率 | ⭐⭐ | 熟人社交环境，公开商业评论少 |
| 商业意向强度 | ⭐⭐ | 私域场景，公开评论商业意图弱 |
| 数据获取难度 | ⭐ | 无公开 API，几乎不可自动化 |

**采集优先级：P2（预留，不主动启动）**

## 三、账号发现模型

### 3.1 核心原则

**不要只靠账号名称判断。** 账号名称可以伪装（如"成都光伏专家"实际是外地品牌）。真正判断依据是：

| 判断依据 | 权重 | 说明 |
|---------|:----:|------|
| **账号内容** | 40% | 视频/笔记的实际主题——安装实拍？科普？广告？ |
| **视频标题** | 25% | 包含了哪些场景、区域、关键词 |
| **评论区内容** | 25% | 评论中客户密度最高——询价/咨询/要联系方式的频率 |
| **账号名称/简介** | 10% | 辅助参考，不独立作为判断依据 |

> 引用：`COMPETITOR_SCORE_RULE.md` §二 六维评分（业务匹配 30 + 城市家庭光伏 20 + 别墅权重 15 + 区域 15 + 需求价值 10 + 活跃度 10）。

### 3.2 四级竞品分类

#### 一级：全国光伏品牌

| 属性 | 说明 |
|------|------|
| 典型特征 | 全国性光伏安装品牌官方号，内容以品牌展示和案例分享为主 |
| 城市客户信号 | 🟡 中。覆盖面广，但评论分散，需按 IP 属地过滤川渝黔 |
| 发现方式 | 关键词搜索（光伏+安装+品牌名），抖音/小红书品牌官方号 |
| 采集策略 | 仅采集 IP 属地 = 川渝黔的评论 |

#### 二级：区域安装商

| 属性 | 说明 |
|------|------|
| 典型特征 | 川渝黔本地光伏安装公司/团队账号，服务范围明确在核心市场 |
| 城市客户信号 | 🔴 极高。"成都这边怎么联系""重庆能上门看吗"类评论密集 |
| 发现方式 | 城市词 + 光伏词搜索（"成都 光伏安装""重庆 家庭光伏"） |
| 采集策略 | 全量采集，最高频率 |

#### 三级：城市小区案例账号

| 属性 | 说明 |
|------|------|
| 典型特征 | 以高端住宅光伏案例展示为主（别墅实拍、叠拼安装、阳光房改造），内容本身筛选了受众 |
| 城市客户信号 | 🔴 极高。"我家也是别墅""这个多少钱""来我家看看" |
| 发现方式 | 房屋词 + 光伏词搜索（"别墅 光伏""阳光房 光伏""叠拼 光伏"） |
| 采集策略 | 全量采集，最高频率 |

#### 四级：装修/别墅/住宅改造账号

| 属性 | 说明 |
|------|------|
| 典型特征 | 非纯光伏账号，以高端住宅装修/别墅改造/阳光房设计为主题，内容中交叉涉及光伏 |
| 城市客户信号 | 🟡 中高。受众为有装修预算的城市业主，光伏为交叉需求 |
| 发现方式 | 房屋词 + 装修词搜索（"别墅 装修""阳光房 改造""花园洋房 设计"） |
| 采集策略 | 仅采集含"光伏""太阳能""新能源"关键词的视频评论 |

## 四、视频发现模型

### 4.1 哪些视频进入评论采集

视频标题和内容主题是判断的核心依据。以下六类视频的评论区是城市小区光伏客户的集中来源：

| 视频类型 | 标题特征示例 | 城市客户信号 | 采集方式 |
|---------|------------|:----------:|---------|
| **光伏安装案例** | "成都别墅光伏安装实拍""20kW系统并网成功" | 🔴 极高 | 全量评论 |
| **屋顶改造** | "屋顶光伏改造全过程""露台做光伏阳光房" | 🔴 极高 | 全量评论 |
| **家庭能源** | "家庭光伏一年发了多少电""我家电费从1000到100" | 🟠 高 | 全量评论 |
| **住宅装修** | "别墅装修含光伏设计""花园洋房新能源改造" | 🟡 中高 | 仅光伏相关评论 |
| **收益分析** | "光伏回本要多久""安装光伏到底划不划算" | 🟠 高 | 全量评论 |
| **安装价格** | "装一套光伏要多少钱""不同房屋类型光伏价格对比" | 🟠 高 | 全量评论 |

### 4.2 视频发现流程

```
Step 1: 获取竞品账号最近 N 条视频（默认 30 条）
    │
    ▼
Step 2: AI 分析视频标题
    ├── 命中六类主题 → 标记 video_type，进入采集
    ├── 命中房屋关键词（别墅/叠拼/阳光房/大平层）→ housing_match，全量采集
    └── 无关视频 → 跳过
    │
    ▼
Step 3: 进入采集的视频
    ├── housing_match = true → 全量评论（不限时间）
    └── housing_match = false → 仅近 7 天评论
```

## 五、评论采集规则

### 5.1 核心原则

**不是关键词命中采集。** 采集原始评论，交给 AI 判断。

### 5.2 错误方式 vs 正确方式

| 错误方式 | 正确方式 |
|---------|---------|
| 只采集命中"成都光伏"的评论 | "我是成都的，可以安装吗？" → 采集，让 AI 判断 |
| 只采集命中"多少钱"的评论 | "我家屋顶不知道能不能装" → 采集，让 AI 判断 |
| 只采集命中"别墅"的评论 | "你家这个装下来贵不贵" → 采集，让 AI 判断 |
| 采集阶段就判断客户价值 | "安装多久回本" → 采集，让 AI 判断 |

### 5.3 采集原则

| 原则 | 说明 |
|------|------|
| **全量采集** | 进入采集的视频，评论区不预设过滤，全部采集 |
| **不预判价值** | collector 不判断"这条评论是否有价值"，全部交给下游意图模型 |
| **保留语境** | 每条评论必须关联原始视频标题，供下游 AI 理解语境 |
| **保留原始文本** | 不修改、不简化评论内容，保留用户的完整表达 |

### 5.4 采集阶段唯一过滤

仅过滤以下无法分析的无效数据：

| 过滤条件 | 原因 |
|---------|------|
| 评论内容为空 | 无分析价值 |
| 评论内容仅 emoji（无文字） | 无语义内容 |
| 完全重复采集（同 comment_id 已存在） | 去重 |

以下情况**不过滤**：

| 评论类型 | 不过滤的原因 |
|---------|------------|
| 看似与光伏无关 | AI 可能通过语义发现隐含意图 |
| 来自农村 IP | 遵循 CUSTOMER_SCORE_MODEL §8 原则 |
| "了解一下""看看" | 可能是观望型客户，下游判断 |
| 含联系方式/广告嫌疑 | 下游意图模型负责识别 |
| 负面/质疑评论 | 下游真实性判断需要全量数据 |

## 六、输出数据结构

### 6.1 输出字段定义

必须兼容 `comment_analyzer` 输入（引用 `comment_analyzer/agent.yml` 和 `comment_schema.md`）。

```json
{
  "platform": "string          // douyin / xiaohongshu / bilibili / kuaishou / wechat_video",
  "account": {
    "account_id": "string       // 平台账号 ID",
    "account_name": "string     // 账号昵称",
    "account_grade": "string    // S / A / B（竞品等级）"
  },
  "video_id": "string           // 视频/笔记唯一 ID",
  "video_title": "string        // 视频标题（下游语义理解的上下文）",
  "comment_id": "string         // 评论唯一 ID（平台 + 视频ID + 评论ID 拼接）",
  "comment_text": "string       // 用户原始评论文本（完整保留，不修改）",
  "publish_time": "datetime     // 评论发布时间（YYYY-MM-DD HH:MM:SS）",
  "region_hint": {
    "ip_location": "string      // 平台公开 IP 属地",
    "text_location": "string    // 评论文本中提及的地点（如有）"
  },
  "keyword_source": "string     // 触发本次采集的关键词组合",
  "collected_time": "datetime   // 采集时间",
  "source_url": "string         // 评论所在视频链接"
}
```

### 6.2 与 comment_analyzer 输入兼容性

| collector 输出字段 | comment_analyzer 消费字段 | comment_schema 对应 |
|-------------------|-------------------------|-------------------|
| `platform` | 透传 | `platform` |
| `comment_text` | 输入原文 | `content` |
| `comment_id` | 透传 | `id` |
| `publish_time` | 时间判断 | `create_time` |
| `region_hint.ip_location` | 区域识别 | `location` |
| `video_title` | 上下文 | `video_title` |
| `source_url` | 溯源 | `source_url` |
| `keyword_source` | 透传 | `keyword` |

### 6.3 输出目录

```
02_DATA/raw/
├── douyin/
│   └── YYYY-MM-DD/
│       └── batch_HHh_NNN.json
├── xiaohongshu/
├── bilibili/
├── kuaishou/
└── wechat_video/
```

## 七、数据过滤规则

### 7.1 采集阶段：只过滤垃圾数据

| 过滤 | 条件 | 处理 |
|:--:|------|------|
| ✅ | 评论内容为空 | 丢弃 |
| ✅ | 评论内容仅 emoji（无文字） | 丢弃 |
| ✅ | 重复采集（同 comment_id） | 去重，保留最新 |

### 7.2 采集阶段：不做的事情

| 不做 | 具体 | 原因 |
|:--:|------|------|
| ❌ | 不判断客户价值 | "这个有价值/没价值" → 下游 AI 判断 |
| ❌ | 不判断客户等级 | "这可能是 S 级客户" → lead_scoring_agent 判断 |
| ❌ | 不判断农村/城市 | "这是农村客户，跳过" → CUSTOMER_SCORE_MODEL §8 禁止 |
| ❌ | 不判断意向高低 | "这个意向高/低" → intent_model 判断 |
| ❌ | 不修改评论内容 | "把方言改成普通话" → 保持原始文本 |

### 7.3 核心规则

> 采集阶段不提前判断客户价值。
>
> 所有评论全量进入下游。
>
> AI 判断哪些是高价值客户。
>
> 不是采集阶段筛选。

### 8.3 与 10_AI_AUTOMATION_ENGINE 的连接
